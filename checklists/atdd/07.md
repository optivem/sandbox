**Shared infrastructure:**

- [ ] `UseCase` interface is defined
- [ ] `BaseUseCase` holds driver and context
- [ ] `UseCaseContext` manages param and result alias maps
- [ ] `VoidVerification`, `ResponseVerification`, and `ErrorVerification` are defined


**App DSL:**

- [ ] `AppDsl` is the entry point with one accessor per domain
- [ ] Each accessor creates the domain DSL lazily


**Shop DSL and use cases:**

- [ ] `ShopDsl` has one factory method per use case
- [ ] Each use case that returns data has a paired `*Verification` class
- [ ] Use cases with no data response use `VoidVerification`


**External system DSLs:**

- [ ] Each external system has a DSL class following the same pattern as `ShopDsl`


**Tests:**

- [ ] `BaseAppDslTest` exposes `app` and no longer exposes raw drivers
- [ ] All smoke tests use `app.shop().goToShop().execute().shouldSucceed()` style
- [ ] All E2E tests use DSL use case calls with no manual request construction
- [ ] Aliases are used to thread system-generated values between use cases
